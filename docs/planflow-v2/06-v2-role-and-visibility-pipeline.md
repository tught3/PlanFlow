# PlanFlow V2 Role and Visibility Pipeline

## 1. V2 최종 제품 정의

PlanFlow V2는 팀 운영 플랫폼 전체가 아니라, 기존 개인 AI 일정관리 앱을 유지하면서 팀장에게 간단한 팀 일정 관리/조회 기능을 추가하는 버전이다.

핵심은 다음과 같다.

- 기본은 개인 일정관리
- 팀 기능은 팀에 소속된 사용자에게만 표시
- 팀장에게만 팀 대시보드와 팀원별 조회 기능 제공
- 팀원은 자신이 속한 팀 일정과 본인 액션아이템만 확인

## 2. 사용자 상태 구분

### A. 개인 사용자

- 팀에 속하지 않음
- 기존 PlanFlow 개인 일정만 사용
- 팀 메뉴/팀 일정/팀 대시보드 노출 안 함

### B. 팀원

- 하나 이상의 팀에 초대되어 수락한 사용자
- 개인 일정은 그대로 사용
- 소속 팀의 공유 일정 조회 가능
- 오늘 팀 일정 확인 가능
- 오늘 회의 확인 가능
- 내 액션아이템 확인 가능
- 팀 대시보드/팀원별 일정 조회는 불가

### C. 팀장

- 자신이 만든 팀 또는 팀장 권한이 부여된 팀 보유
- 팀 이름 생성/수정 가능
- 팀원 초대/제거 가능
- 팀 공유 일정 생성/수정/삭제 가능
- 팀원별 팀 일정 조회 가능
- 요일별 팀 일정 조회 가능
- 회의록 작성/AI 요약/액션아이템 승인 가능
- 팀 대시보드 조회 가능

### D. 다중 팀 사용자

- 한 사람이 여러 팀을 만들 수 있음
- 한 사람이 여러 팀에 팀원으로 들어갈 수 있음
- 팀 전환 UI 필요
- 팀별 데이터는 반드시 `team_id` 기준으로 분리

## 3. 역할 구조

V2 MVP에서는 역할을 단순화한다.

- `team_leader`
- `team_member`

나중에 확장 후보:

- `department_manager`
- `admin`
- `viewer`

단, V2에서는 본부장/관리자 구조를 과하게 구현하지 않는다.

## 4. 팀 생성 파이프라인

사용자 → 팀 만들기 → 팀 이름 입력 → `teams` 생성 → 생성자를 `team_leader`로 `team_members`에 등록 → 팀 홈 진입

필수 조건:

- 한 사용자가 여러 팀 생성 가능
- 팀 이름 수정 가능
- 팀 삭제는 V2에서는 보류하거나 신중하게 처리

## 5. 팀 초대 파이프라인

팀장 → 팀원 초대 → 이메일 입력 → `team_invites` 생성 → 초대받은 사용자 확인 → 수락 → `team_members`에 `team_member`로 등록

주의:

- 초대 수락 전에는 팀 데이터 조회 불가
- 초대 만료/취소 상태 필요
- 이미 팀원인 이메일 중복 초대 방지

## 6. 팀 일정 파이프라인

팀장 또는 허용된 팀원 → 팀 일정 생성 → `team_events` 저장 → 해당 `team_id`의 모든 팀원이 조회 가능

표시 규칙:

- 개인 일정은 `events`에서 조회
- 팀 일정은 `team_events`에서 조회
- 캘린더 UI에서는 두 데이터를 병합 표시
- DB는 절대 병합하지 않음
- 팀 일정에는 `팀` 배지와 팀 이름 표시

## 7. 팀장 대시보드 파이프라인

팀장 → 팀 선택 → 팀 대시보드 진입

표시 정보:

- 오늘 팀 일정
- 이번 주 팀 일정
- 요일별 팀 일정
- 팀원별 팀 일정
- 오늘 회의
- 미확정 액션아이템
- 팀원별 미완료 액션아이템 수

단, V2에서는 통계/성과분석/AI 코칭은 제외한다.

## 8. 팀원 화면 노출 파이프라인

팀원 → 앱 실행

표시:

- 개인 일정
- 소속 팀 일정
- 오늘 회의
- 내 액션아이템

미표시:

- 팀원별 전체 일정
- 팀 대시보드
- 팀 관리
- 다른 팀원의 액션아이템 상세

## 9. 회의록/액션아이템 파이프라인

팀 일정 또는 회의 메뉴 → 회의록 작성 → `meeting_notes` 저장 → AI 요약 → `meeting_action_items` 후보 생성 → 팀장 검토 → 담당자/마감일 확정 → `task` 또는 `confirmed_action_item` 생성

주의:

- AI가 만든 액션아이템은 바로 확정 업무가 아님
- 팀장 승인 전에는 후보 상태
- 담당자는 `team_members` 기준으로 선택
- 팀원은 자신에게 할당된 항목만 명확히 확인

## 11. 데이터 분리 원칙

반드시 다음 원칙을 지킨다.

- 개인 일정은 `events`
- 팀 일정은 `team_events`
- 팀 구성은 `teams` / `team_members`
- 초대는 `team_invites`
- 회의록은 `meeting_notes`
- 액션아이템은 `meeting_action_items`
- 간단 업무는 `tasks` 또는 `confirmed_action_items`

개인 `events`에 `team_id`를 붙이는 방식은 사용하지 않는다.

## 12. 최종 결론

V2는 팀 운영 플랫폼이 아니라, 개인 일정관리 앱에 팀장용 팀 일정 조회/공유/회의 후속조치 기능을 얹는 구조다.

따라서 구현 전 반드시 다음이 확정되어야 한다.

- 사용자 상태별 화면 노출 조건
- `team_id` 기준 데이터 분리
- 팀장/팀원 권한 차이
- 팀 일정과 개인 일정 오버레이 방식
- 액션아이템 후보/확정 분리
